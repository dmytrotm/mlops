# Інструкції з локального запуску та перевірки платформи (ЛР 3, 4, 5)

Цей посібник допоможе розгорнути проєкт локально, перевірити гіперпараметричну оптимізацію (ЛР3), протестувати механізми CI/CD-gate (ЛР4) та запустити Airflow оркестратор у Docker (ЛР5).

## Крок 1: Базове налаштування оточення

Створимо та активуємо віртуальне середовище Python 3.11+:

```bash
python3 -m venv venv
source venv/bin/activate  # Для macOS/Linux
# venv\Scripts\activate   # Для Windows
```

Встановимо всі необхідні залежності:

```bash
pip install --upgrade pip
pip install -r requirements.txt
pip install pytest "apache-airflow==2.9.2"  # Для локального тестування DAG-ів
```

## Крок 2: Перевірка ЛР3 (Optuna HPO та MLflow)

В рамках 3-ї лабораторної налаштована оптимізація гіперпараметрів (HPO) за допомогою фреймворку Optuna з логуванням у локальну базу даних SQLite для MLflow.

1. **Запуск HPO оптимізації через Hydra CLI:**
   ```bash
   python src/optimize.py hpo.n_trials=10 hpo.sampler="tpe"
   ```
   *Дана команда перезаписує дефолтну конфігурацію і налаштовує TPE-семплінг на 10 ітерацій. Пайплайн натренує модель з різними параметрами та збереже їх у локальний `mlflow.db`.*

2. **Запуск MLflow Server для перегляду звітів:**
   ```bash
   mlflow ui --backend-store-uri sqlite:///mlflow.db
   ```
   Після старту сервера, перейдіть в браузері за посиланням `http://127.0.0.1:5000/`. Обравши експеримент `HPO_Lab3`, ви зможете побачити батьківський запуск та дочірні (Nested) запуски (trials), графік оптимізації, артефакти моделі та конфігурації.

## Крок 3: Перевірка ЛР4 (CI/CD Тестування та Quality Gate)

Структура автоматизованого тестування поділяється на дві частини:
- **Pre-train (`test_data.py`)**: Швидка перевірка сирих даних та валідація їх схеми до початку навчання.
- **Post-train (`test_model.py`)**: Перевірка артефактів після навчання (метрики, `model.pkl`) та перевірка порогів (Quality Gate).

1. **Виконання повного тестового набору:**
   Для перевірки даних, моделей, а також цілісності Airflow DAG-ів, введіть команду:
   ```bash
   pytest tests/ -v --tb=short
   ```

2. **Кастомний запуск Quality Gate (зміна порогів):**
   Ви можете емулювати падіння `Quality Gate` в CI. Якщо задати поріг F1 дуже високим `0.99`, то тест провалиться:
   ```bash
   F1_THRESHOLD=0.99 pytest tests/test_model.py::TestQualityGate::test_quality_gate_f1 -v
   ```

> *Окрім локальних тестів, перевірки повністю вбудовані у GitHub Actions (файл `.github/workflows/cml.yaml`), які при Pull Request генерують CML-report з Baseline-порівнянням.*

## Крок 4: Перевірка ЛР5 (Airflow та Docker)

Запуск повноцінного автоматизованого оркестратора (безперервне навчання/CT) відбувається у власному ізольованому контейнерованому середовищі.

1. **Збірка образу та підняття Airflow кластеру:**
   Буде використано оптимізований Multi-stage Dockerfile:
   ```bash
   docker-compose up -d --build
   ```

2. **Ініціалізація та доступ:**
   - Почекайте хвилину, поки Airflow Scheduler та Webserver ініціалізують PostgresDB.
   - Відкрийте сторінку `http://localhost:8080/` (стандартний логін і пароль `airflow` / `airflow`).
   
3. **Запуск Pipelines:**
   - Знайдіть DAG під назвою `ml_training_pipeline`.
   - Активуйте його перемикачем у стан `Unpaused` і натисніть "Trigger DAG" (Play).
   - Ви зможете в реальному часі побачити, як Airflow перевіряє `FileSensor`, запускає DVC-команди: `prepare_data`, `train_model`, ухвалює рішення в `evaluate_and_branch`, і нарешті, якщо модель проходить поріг з Accuracy, — виконує `register_model` до Model Registry.

Для зупинки контейнерів:
```bash
docker-compose down -v
```
